NexusAID — Universal Humanitarian Payment Rails
A donor anywhere in the world sends fiat in any currency. It enters XRPL, auto-converts to RLUSD via the native DEX, and lands in a recipient's Humanitarian ID wallet — accessible via a web dashboard or SMS, no smartphone required. NGOs manage disbursements through a dashboard. Recipients withdraw to local mobile money or cash out at partner agents.
Ripple / XRPL track RLUSD stablecoin Humanitarian ID SMS access NGO dashboard
Why XRPL is perfect for this — not just allowed, actually ideal
XRPL's native DEX handles cross-currency conversion in a single transaction with no slippage risk and near-zero fees (~$0.0002). Trust lines let you issue RLUSD to recipient wallets without them needing to hold XRP first. Payment paths automatically find the best conversion route. This is infrastructure built exactly for this use case — you're not forcing it, you're using it as designed.
The Humanitarian ID — the core innovation
Every recipient gets a Humanitarian ID: a lightweight XRPL wallet created and custodied by the NGO, with a human-readable identifier (e.g. HID-2847-KE). The recipient doesn't need to understand crypto. They receive an SMS with their balance and a PIN. They can spend it at partner merchants, withdraw to M-Pesa, or receive additional disbursements. The ID is portable — it moves with the person, not the organization.
What makes this real-world ready for judges
SMS access means recipients in low-connectivity regions with basic feature phones can use this. The NGO dashboard gives aid organizations actual operational tools — not just a wallet. RLUSD as the settlement currency removes volatility risk for recipients. And every disbursement is on-chain and auditable, which is exactly what institutional donors (governments, foundations) require before releasing funds.

End-to-end flow — donor fiat to recipient RLUSD
1
Donor visits donation portal and pays fiat
nexusaid.org/donate — Next.js frontend. Donor selects cause and amount ($10, $50, $100). Stripe Checkout handles card processing. On success, Stripe webhook fires to your backend with the confirmed fiat amount. Donor receives an email with a "Track your donation" link.
Stripe Next.js
2
Backend triggers XRPL on-ramp
Your Node.js backend receives the Stripe webhook. It calls a fiat-to-XRP on-ramp provider (Transak or similar — or simulate for the demo). The backend holds a "gateway wallet" on XRPL that receives the XRP equivalent of the fiat amount. This is the bridge between the traditional finance world and XRPL.
XRPL Gateway Wallet Transak API
3
XRPL DEX auto-converts XRP → RLUSD
The backend submits an OfferCreate transaction on the XRPL native DEX to swap XRP for RLUSD. Because XRPL uses pathfinding, this happens atomically in one transaction — no separate swap step needed. The RLUSD lands directly in the NGO's distribution wallet. Fee: ~$0.0002. Settlement: 3–5 seconds.
XRPL DEX OfferCreate RLUSD Trust Line
4
NGO dashboard routes funds to recipient Humanitarian IDs
NGO operators see incoming funds in their dashboard in real-time via Socket.io. They can set automatic disbursement rules ("distribute equally to all active recipients in Kenya") or manually assign amounts. The dashboard submits XRPL Payment transactions from the NGO distribution wallet to each recipient's Humanitarian ID wallet.
NGO Dashboard XRPL Payment tx
5
Recipient notified via SMS
When the XRPL payment to their wallet confirms, your backend listens for the ledger event and triggers a Twilio SMS to the recipient's registered phone number: "NexusAID: You received $12.50 RLUSD. New balance: $47.20. Reply BAL for balance, SEND [amount] [phone] to transfer, HELP for options." Works on any phone — no smartphone, no app, no internet needed.
Twilio SMS XRPL ledger listener
6
Recipient accesses funds via SMS commands or web
SMS: Reply "BAL" → backend queries XRPL and replies with balance. Reply "SEND 5 +254712345678" → backend executes XRPL payment from their wallet to the recipient's wallet. Web dashboard (for smartphone users): shows balance, transaction history, QR code for in-person payments at partner merchants. All wallet operations are custodied by the backend — recipient authenticates with PIN only.
Twilio SMS commands Next.js recipient portal
7
Donor tracks their donation end-to-end
The "Track your donation" link shows the donor a live timeline: "Donation received → Converted to RLUSD on XRPL [tx hash] → Disbursed to recipient HID-2847-KE [tx hash] → Recipient notified." Every step is an XRPL transaction — immutable, auditable, publicly verifiable. This is the transparency story that wins institutional donor trust.

XRPL concepts you must implement
1. Wallets and account structure
Four wallet types on XRPL testnet:

Gateway wallet — receives XRP from on-ramp, holds trust line to issue RLUSD. This is the "bank" of the system.

NGO distribution wallet — receives RLUSD from gateway, disbursed to recipient wallets. NGO operators control this via the dashboard.

Humanitarian ID wallets — one per recipient, created by the backend. Custodied (backend holds the secret key, encrypted). Funded with the XRPL minimum reserve (10 XRP) on creation.

Donor tracking wallet — optional, used to give donors a unique address to track their specific contribution through the system.
2. Trust lines — how RLUSD works
RLUSD on XRPL requires trust lines. Every recipient wallet must establish a trust line to the RLUSD issuer (Ripple's official RLUSD issuer on testnet) before it can receive RLUSD. Your backend does this automatically when creating a Humanitarian ID wallet:

const trustSetTx = { TransactionType: "TrustSet", Account: recipientWallet.address, LimitAmount: { currency: "RLUSD", issuer: RLUSD_ISSUER, value: "10000" } }

Sign and submit this immediately after wallet creation. The wallet can now receive RLUSD.
3. XRPL native DEX — XRP to RLUSD conversion
Use Payment with path-finding rather than OfferCreate for simplicity. XRPL will automatically route XRP through the DEX to deliver RLUSD:

const paymentTx = {
TransactionType: "Payment",
Account: gatewayWallet.address,
Destination: ngoWallet.address,
Amount: { currency: "RLUSD", issuer: RLUSD_ISSUER, value: "50.00" },
SendMax: xrpToDrops("55"), // max XRP willing to spend
Paths: [...] // from rippled pathfinding
}

Use client.request({ command: "ripple_path_find", ... }) to get paths before submitting.
4. Ledger subscription — real-time event listening
Subscribe to account transactions to trigger SMS notifications when payments arrive:

await client.request({ command: "subscribe", accounts: [recipientWallet.address] })

client.on("transaction", (tx) => {
if (tx.transaction.Destination === recipientWallet.address) {
sendSMS(recipientPhone, `You received ${amount} RLUSD`);
}
})

Subscribe to all active recipient wallet addresses on backend startup. This is how SMS notifications fire instantly when funds arrive.
5. npm package to use
npm install xrpl — the official Ripple SDK. Use Client for connections, Wallet for key management, and the transaction helpers for building Payment, TrustSet, and OfferCreate transactions. Connect to XRPL testnet: wss://s.altnet.rippletest.net:51233. Get testnet XRP from the faucet: client.fundWallet() — it gives you 1,000 test XRP instantly.
Five components — what each one is and does
1. Donation Portal
Next.js — new build
Landing page: cause selection, impact stats, "donate now" CTA
Donate page: amount selector, Stripe Checkout embed
Tracker page: /track/[donationId] — live timeline of donation journey with XRPL tx hashes
Socket.io: real-time update when donation converts and disburses
2. Backend API
Node.js / Express — core
Stripe webhook handler: receives payment confirmation, triggers XRPL flow
XRPL service: wallet creation, trust lines, payments, DEX swaps, ledger subscription
SMS service: Twilio webhook receiver, command parser (BAL / SEND / HELP), response sender
Humanitarian ID service: create, fund, and manage recipient wallets
Socket.io: real-time events to NGO dashboard and donor tracker
3. NGO Dashboard
Next.js — new build
Fund overview: incoming donations, total in treasury, disbursed to date
Recipient registry: table of all Humanitarian IDs, balances, last activity
Disburse funds: select recipients, enter amounts, confirm — submits XRPL payments
Audit log: every XRPL tx hash, timestamped, exportable
Live feed: Socket.io stream of incoming donations
4. SMS Gateway
Twilio — new service
Inbound: Twilio webhook → backend parses SMS command
BAL: query XRPL for wallet balance, reply with formatted amount
SEND [amt] [phone]: look up recipient by phone, submit XRPL Payment
HELP: reply with available commands
Outbound: notification SMS on any incoming payment to wallet
5. Recipient Web Portal
Next.js — lightweight
Login: enter Humanitarian ID + PIN (no wallet, no crypto knowledge needed)
Balance: current RLUSD balance in large text + local currency equivalent
History: list of all incoming and outgoing transactions
Send: transfer to another HID by phone number or ID
QR code: for in-person payments at partner merchants
Reuse from NexusPay / TapGuard
The NFC payment flow from TapGuard can be the "partner merchant" piece — a recipient shows their QR code or taps their phone at a merchant terminal, which triggers an XRPL payment from their Humanitarian ID wallet. This is optional but if you have the NFC code, it's an extremely powerful demo addition — showing aid funds being spent in the real economy, not just sitting in a wallet.
